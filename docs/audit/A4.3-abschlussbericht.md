# A4.3 Abschlussbericht — PermissionKey-Type-Drift (Typ-Stabilisierung)

**Datum:** 2026-06-20  
**Branch:** main  
**Scope:** P0 Mini-Gruppe A4.3 — PermissionKey/Role/Guard TYPE-Struktur analysieren und minimal stabilisieren (keine Gruppe-B-Fachfixes)

---

## Kurzfazit

A4.3 hat **129 PermissionKey-Zuweisungsfehler auf 0** reduziert und die Gesamtfehlerzahl von **862 auf 713** gesenkt (−149, −17 %). Ursache war **String-Drift**: 38 Permission-Strings waren bereits im Code (Services, Screens, Modul-Konstanten), fehlten aber in der zentralen `PermissionKey`-Union und in `PERMISSION_LABELS`. Minimal-Fix: fehlende Keys ergänzen, Compile-Time-Guard für Label-Abdeckung, falscher `as RoleKey`-Cast in `calendarEventSaveService` entfernt. **Keine** Rollen-Matrix-, RLS-, Route-Gate- oder Intake-Fachlogik geändert.

## Key Findings für Gruppe-B-Vorbereitung

1. **Typ vs. Fach:** Keys existieren jetzt im Typ-System, sind aber für die meisten Rollen in `ROLE_PERMISSIONS` **noch nicht** freigeschaltet → Runtime `can()` bleibt `false` trotz typ-sicherem Code.
2. **Invoice create:** Nutzt `office.invoices.view` mit `as never` — separates Create-Recht fehlt in Union und Matrix.
3. **Zwei Permission-Systeme:** Feingranulare `PermissionKey`-Liste vs. `PermissionAreaKey`/`PermissionAction`-Matrix — Mapping für B klären.
4. **Guards dünn auf Routen:** Fast alle Checks in Screens/Services via `can()` / `enforcePermission`, nicht `RequirePermission`.
5. **DB-Drift-Risiko:** `RolePermission.permissionKey: string` und Supabase-Runtime-Permissions müssen mit Union synchron gehalten werden (Migration/Seed in B).
